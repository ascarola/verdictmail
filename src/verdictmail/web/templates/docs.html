Documentation

VerdictMail — AI Email Threat Analysis Platform  ·  v0.1.0-beta

Overview

VerdictMail is a headless Linux daemon that monitors a Gmail inbox using IMAP IDLE — a push-based protocol that delivers new message notifications in real time without polling. Each incoming message is automatically routed through a multi-stage enrichment and AI analysis pipeline, then acted on without any user intervention.

The platform runs as two systemd services on an Ubuntu 24.04 LXC container:

  • verdictmail.service — the analysis daemon (IMAP IDLE loop + worker threads)
  • verdictmail-web.service — this Flask admin interface (port 80)

Messages are processed concurrently; each worker opens its own dedicated IMAP connection so it never disturbs the IDLE socket used for monitoring.

Processing Pipeline

Every new message passes through the following stages in order:

#StageWhat happens
1 Fetch The worker opens a private IMAP connection and fetches the full raw message using BODY.PEEK[], which retrieves message bytes without marking the message as read (\Seen).
2 Parse The raw RFC 822 bytes are decoded into a structured object: sender address, display name, subject, all headers, plain-text body, HTML body, and a deduplicated list of URLs.
2a Whitelist check The sender and subject are compared against your whitelist rules. If a match is found the message is immediately passed with no further processing — skipping enrichment and AI entirely. See Whitelist.
3 Enrich Gathers objective signals about the sender and message: SPF / DKIM / DMARC validation, domain age (WHOIS), DNSBL reputation checks, display-name spoofing detection, and URL expansion for shortened links.
4 AI Analysis All enrichment signals are packaged into a structured prompt and sent to the configured AI provider (OpenAI, Anthropic, or a local Ollama model). The model returns a JSON object containing a threat level, confidence score, threat type list, and human-readable reasoning.
5 Decide The decision engine applies configurable confidence thresholds to the AI result and selects a final action: pass, flag, or move_to_junk. See Decision Logic.
6 IMAP Action The action is applied using the worker's dedicated IMAP connection. See Actions.
7 Audit Log A complete record — sender, subject, all signal values, AI result, final action, processing time — is written to the SQLite database regardless of outcome, including errors. See Audit Log.
Threat Levels

The AI model classifies every message into one of five threat levels:

LevelMeaningDefault action
none Clean mail — no suspicious signals detected. pass
low Minor signals present (e.g. bulk sender, marketing mail) but not indicative of a threat. Google's own spam filters are better suited to handle these. pass
medium Notable signals that warrant attention — possible phishing attempt, suspicious domain, or unusual sending patterns. flag if confidence ≥ flag threshold
high Strong indicators of a malicious message — likely phishing, BEC, malware delivery, or credential harvesting. flag or junk depending on confidence
critical Near-certain threat. Model has very high confidence this is a malicious message. move_to_junk always
Actions

VerdictMail takes exactly one of three actions on each message:

PASS
No IMAP changes are made. The message remains in your inbox exactly as delivered. This is the outcome for clean mail, low-threat mail, and all whitelisted senders.
FLAG
Sets the custom IMAP keyword $VerdictMail-Suspect on the message. The message stays in your inbox.

Note: Gmail's web UI does not render custom IMAP keywords as visible labels, so this flag is not visually apparent in Gmail — but it is queryable via IMAP and is recorded in the audit log.
MOVE_TO_JUNK
Copies the message to the configured junk folder (default: [Gmail]/Spam), then deletes and expunges the original from INBOX. The copy step is confirmed before deletion to prevent message loss on error. The destination is set via imap.junk_folder in the config.
Decision Logic

The decision engine combines the AI's threat level, confidence score (0.0–1.0), and recommended action against two configurable thresholds:

  • Flag threshold (default 0.55) — minimum confidence to FLAG a medium or high threat.
  • Junk threshold (default 0.80) — minimum confidence to MOVE_TO_JUNK a high threat.

Rules are evaluated in priority order — the first matching rule wins:

PriorityConditionAction
1 threat_level == "none" or AI recommended action is pass pass
2 threat_level == "low" pass
3 threat_level == "critical" move_to_junk
4 threat_level == "high" and confidence ≥ junk_threshold move_to_junk
5 threat_level == "high" and confidence ≥ flag_threshold and AI recommends quarantine / junk / block move_to_junk
6 threat_level in ("medium", "high") and confidence ≥ flag_threshold flag
7 Everything else pass

Thresholds are adjusted on the Configuration page under thresholds.flag and thresholds.junk. A daemon restart is required for changes to take effect.

Whitelist

The whitelist lets you exempt trusted senders from analysis entirely. Whitelisted messages are passed immediately after parsing — skipping enrichment, AI, and IMAP actions — and logged with action=pass, threat_level=none, confidence=1.0, and a reasoning note of "Whitelisted: <comment>".

Rule fields

Use sender to trust one specific address, or sender_domain to trust all addresses at a domain. Either can be combined with subject_contains to narrow the match.

FieldMatch typeExample
sender Exact email address (case-insensitive) alerts@mybank.com
sender_domain Domain portion of From address (case-insensitive) example.com
subject_contains Case-insensitive substring of Subject Weekly digest
comment Human-readable label (not used for matching) Trusted sender
Matching logic
  • A rule must specify at least one of sender, sender_domain, or subject_contains.
  • If a rule specifies multiple fields, all must match (AND logic).
  • Rules are tested in order; the first matching rule wins (OR logic across rules).
  • Whitelist changes take effect after a daemon restart.
Example combinations
whitelist:
  enabled: true
  rules:
    # Exact sender — trust all mail from this address
    - sender: nas-alerts@example.com
      comment: NAS device notifications

    # Domain — trust all mail from this domain
    - sender_domain: example.com
      comment: All internal example.com addresses

    # Sender + subject — only trust one type of mail from a noisy sender
    - sender: newsletter@bigretailer.com
      subject_contains: "order confirmation"
      comment: Order confirmations only (not marketing)

Whitelist rules are managed on the Whitelist page.

Audit Log

Every processed message generates a record in the SQLite database at /var/log/verdictmail/verdictmail.db, regardless of outcome (including errors and whitelisted messages). Records are viewable on the Audit Log page.

ColumnDescription
timestampUTC ISO-8601 time the record was written.
senderBare From address (lowercased).
subjectMessage subject.
threat_levelAI classification: none / low / medium / high / critical. "unknown" if the pipeline errored before AI.
confidenceAI confidence score, 0.0–1.0. Set to 1.0 for whitelisted messages.
action_takenFinal action: pass, flag, move_to_junk, or error:<action> if the IMAP step failed.
reasoningAI's explanation. For whitelisted messages: "Whitelisted: <comment>".
model_nameAI model used (e.g. gpt-4o-mini). Set to "whitelist" for whitelisted messages.
processing_msTotal wall-clock time for the pipeline in milliseconds.
signalsJSON object of all enrichment signals (SPF, DKIM, DMARC, DNSBL hits, domain age, etc.).
Gmail & IMAP Notes

VerdictMail fetches messages using BODY.PEEK[] (RFC 3501), which retrieves the full message without setting the \Seen flag. The older RFC822 IMAP item would mark messages as read — and RFC822.PEEK is not supported by Gmail. This means all messages remain unread in Gmail after VerdictMail processes them.

The $VerdictMail-Suspect IMAP keyword is set at the IMAP protocol level. Gmail's web interface only renders labels it knows about (stars, categories, user-created labels) and silently ignores custom IMAP keywords. The flag is visible to any standard IMAP client and is always recorded in the audit log.

Gmail (particularly Google Workspace accounts) automatically tags messages from outside your domain with an "External" indicator. VerdictMail has no involvement in this — it is a native Gmail/Workspace feature.

When a worker connection touches the monitored folder (e.g. flagging a message), Gmail pushes an unsolicited FETCH FLAGS response to the IDLE connection. VerdictMail silently ignores these "unexpected response" events — they are normal Gmail behavior, not errors.

Some legitimate senders (especially small businesses and self-hosted mail servers) use outgoing IPs that appear on DNSBL lists due to other tenants on the same hosting provider. A DNSBL hit alone does not cause VerdictMail to take action — it is one signal among many that the AI weighs holistically. If a trusted sender consistently gets flagged due to DNSBL, add them to the Whitelist.
Operations
Service management

The daemon can be started, stopped, and restarted directly from the Dashboard. VerdictMail detects its runtime environment on first use and picks the appropriate control strategy automatically:

EnvironmentStrategyStop behaviour
Bare metal / privileged container sudo systemctl Daemon fully stopped; systemd marks the unit inactive. Start/restart via systemctl start.
Unprivileged LXC (e.g. Proxmox) Signal + pause flag Daemon keeps running but ignores incoming messages. A flag file (/var/log/verdictmail/paused) is written; the daemon skips new UIDs while it exists. Resume removes the flag. Incoming messages stay UNSEEN in Gmail and are processed after resuming (picked up by the startup scan on next restart, or as new IDLE events arrive). 
# Check status
systemctl status verdictmail verdictmail-web

# Start / stop / restart daemon
systemctl start verdictmail
systemctl stop verdictmail
systemctl restart verdictmail

# Restart web UI (required after template/route changes)
systemctl restart verdictmail-web

# Follow daemon logs live
journalctl -u verdictmail -f

# Follow web UI logs live
journalctl -u verdictmail-web -f
Inspect the audit database directly
# Most recent 5 decisions
sqlite3 /var/log/verdictmail/verdictmail.db \
  "SELECT id, subject, threat_level,
          printf('%.0f%%', confidence*100),
          action_taken, reasoning
   FROM audit_log ORDER BY id DESC LIMIT 5;"

# Count by action
sqlite3 /var/log/verdictmail/verdictmail.db \
  "SELECT action_taken, COUNT(*) FROM audit_log GROUP BY action_taken;"
Log file

Structured log entries are written to /var/log/verdictmail/verdictmail.log with automatic rotation at 10 MB (5 backups retained). Each processed message emits a summary line: UID <n> done in <ms>ms | threat=<level> confidence=<score> action=<action>
